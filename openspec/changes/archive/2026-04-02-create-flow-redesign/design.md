## Context

当前 `/create` 命令的状态机为：`name → capturing → capture_done → manual_sources → source-path → ingesting → distilling → done`。所有目标统一走 Agent 在线搜索，UNKNOWN_ENTITY 回退到手动数据源选择。问题：私人对象被强制搜索（浪费+污染），且必须有数据源才能完成创建。

参考 colleague-skill 项目的轻量录入模式：仅 3 个自然语言问题即可完成信息收集，数据源为可选增强，纯描述也能生成产物。

## Goals / Non-Goals

**Goals:**
- 用户在 30 秒内完成最小可行创建（名字 + 类型 + 一句话描述）
- 个人灵魂不触发在线搜索，公开灵魂搜索可利用用户描述消歧义
- 建立全局标签体系，为创建、蒸馏、产物导出提供统一的结构化人格元数据
- 无数据源时，仅凭描述和标签就能走完蒸馏流程并生成可用灵魂

**Non-Goals:**
- 进化/纠正机制（后续迭代）
- 导出为 Claude Skill / OpenClaw Skill 格式（后续迭代）
- 新增数据源适配器（微信、飞书等）
- 标签的可视化选择 UI（本次用自然语言输入 + 解析，不做 checkbox 选标签）

## Decisions

### D1: 前置类型选择 — 二分法

**决策**: 用户选择"个人灵魂"或"公开灵魂"两个类型，不做更细的四分类。

**理由**: 用户需要做的核心判断只有一个："这个人的信息在网上能搜到吗？" 公开灵魂内部的子分类（虚构/名人/历史）由 Agent 自动判断，这部分逻辑已有且成熟，无需暴露给用户。

**替代方案**: 四选一（个人/虚构/名人/历史）— 增加选择摩擦，且用户可能纠结边界情况（半虚构角色？）。

### D2: 轻量 3 问录入模型

**决策**: 创建流程收集 3 项信息，每项都是单行自然语言输入：

| 步骤 | 提示 | 可跳过 | 示例 |
|------|------|--------|------|
| Q1 名字 | `灵魂目标:` | ❌ | `老张` / `V` |
| Q2 描述 | `一句话描述 (可跳过):` | ✅ | `我大学室友 程序员 喜欢钓鱼` / `赛博朋克2077主角 女V` |
| Q3 标签 | `性格/印象 (可跳过):` | ✅ | `INTJ 话少 冷幽默 技术洁癖` |

**理由**: 参照 colleague-skill 验证过的模式。自然语言输入比结构化问答摩擦更低，LLM 解析能力完全可以从一句话中提取结构化标签。

**替代方案**: 引导式多问模型（4-5 个段落问题）— 用户输入量太大，测试表明会降低完成率。

### D3: 标签体系 — 定义 + 解析分离

**决策**: 标签体系包含两部分：

1. **标签定义** (`src/tags/taxonomy.ts`): 分类和预置标签的静态定义
2. **标签解析** (`src/tags/parser.ts`): 接收自然语言字符串，通过 LLM 提取匹配的标签 + 自定义标签

标签分类：
```
personality:  MBTI / 九型 / 大五 / 自定义性格词
communication: 沟通风格标签 (直接/委婉/毒舌/冷幽默/...)
values:       价值取向标签 (理性主义/实用主义/...)
behavior:     行为模式标签 (完美主义/拖延/果断/...)
domain:       领域标签 (因类型而异 — 虚构:阵营/来源 | 公众:行业 | 个人:关系)
```

**理由**: 预置标签帮助 LLM 解析有锚点，自定义标签保证灵活性。解析用 LLM 而非规则匹配，因为用户输入形式多样（"话少"="惜字如金"="不爱说话"）。

### D4: 状态机重构

**决策**: 新的 CreateStep 流程：

```
个人灵魂:  type-select → name → description → tags → data-sources → [source-path → ingesting] → distilling → done
公开灵魂:  type-select → name → description → tags → capturing → capture_done → [data-sources → source-path → ingesting] → distilling → done
```

`data-sources` 步骤在两条路径中都是可选的（可以选"跳过"直接进入蒸馏）。

### D5: 纯描述蒸馏路径

**决策**: 当没有任何 SoulChunk 数据时，将用户描述 + 解析出的标签转化为"合成 chunks"喂给现有蒸馏流程。

具体做法：
- Q2 描述 → 生成一个 `type: 'knowledge'` 的 SoulChunk (source: 'user-input')
- Q3 标签解析结果 → 生成一个 `type: 'reflection'` 的 SoulChunk (source: 'user-input')
- 这些合成 chunks 与真实数据 chunks 合并后统一进入 extractor

**理由**: 复用现有蒸馏管线，避免创建全新的"无数据生成"逻辑分支。合成 chunks 质量较低但足以产出基础灵魂文件。

### D6: Agent 搜索接收用户 hint

**决策**: `captureSoul()` 新增可选参数 `hint?: string`（用户的 Q2 描述），拼接到 ROUND1_PROMPT 中辅助分类和消歧义。

**理由**: 用户输入"赛博朋克2077主角 女V"比单独"V"搜索精度高数倍，几乎无实现成本。

## Risks / Trade-offs

**[标签解析质量]** LLM 从一句话提取标签可能不准确 → 解析后展示确认汇总，用户可修改。首版不做高精度要求，后续可以通过进化机制修正。

**[纯描述灵魂质量低]** 仅凭一两句话生成的灵魂必然粗糙 → 定位为"起步灵魂"，通过后续 feed/纠正提升。创建完成时提示用户可以补充数据。

**[状态机复杂度增加]** 从 8 个 step 增加到约 11 个 → 可通过将公共步骤抽取为子组件来管理复杂度。

**[标签体系过度设计]** 首版标签分类可能不够精准 → 只定义分类框架和少量锚点标签，允许自定义标签；后续根据实际使用数据调整。
